# ADR-009: Use Phinx for Database Schema Migrations

## Status

Accepted

## Date

2026-07-09

## Context

The application currently uses web-accessible "FIX scripts" in `/FIX/` to apply
schema changes: column renames, index creation, storage engine conversions, and
new table creation. Each FIX script is a standalone PHP page with a two-phase
UI, progress logging, and a run record written to `fix_script_runs`.

This approach has a fundamental security problem: schema changes should run at
deploy time via CLI, not via browser. A web-accessible script that executes DDL
statements is an unnecessary attack surface, even when protected by an admin
session check. If an attacker can access the page, they can trigger destructive
schema changes.

Beyond the security concern, the FIX script system has no migration tracking
that separates "schema-changing migrations" from "admin utility scripts." Running
a FIX script twice is not safe; the system provides no atomic guarantee that a
given schema change has been applied exactly once. The 243-line FIX script
template provides progress logging and audit trail features that are useful for
administrative utilities, but are over-engineered for the single concern of "apply
this DDL if it has not been applied yet."

The project has approximately 10 historical schema changes and expects moderate,
steady growth: new tables for upcoming features such as the PDF reference library
(ADR-013 introduces FIX script 26 for the `reference_documents` table) and
future enhancements. The scale does not justify a heavyweight enterprise
framework, but the security and tracking gaps in the current approach require a
solution.

The work sits within a broader three-tier model for one-time and recurring
operational scripts:

1. **Formal Migrations** — versioned, tracked-once-per-environment schema
   changes (the subject of this ADR).
2. **Admin UI Tools** — repeatable maintenance utilities run on demand from the
   admin interface, which require human judgment to trigger.
3. **Archive** — completed one-time scripts retained for audit history.

This ADR addresses only the first tier: what mechanism to use for formal schema
migrations.

The broader FIX script restructuring — separating schema migrations from admin
utilities and ad-hoc scripts — is tracked in GitHub issue #595.

### Problem Statement

The application needs a way to:

1. Apply schema changes (DDL/DML) at deploy time, not via browser
2. Track which changes have been applied so they are never run twice
3. Order migrations deterministically and prevent conflicts between contributors
4. Support rollback when a change needs to be reversed

### Why Phinx over a Custom Runner

An earlier draft of this ADR proposed a custom ~100-line runner to avoid a
Composer dependency. That calculus changed once a second, occasional contributor
joined the project. With more than one person authoring migrations, the value of
community documentation (a contributor can learn the tool from its own docs
rather than from a bespoke README), built-in rollback support, and timestamp-based
naming that prevents conflicts outweighs the dependency cost. Phinx pulls in
roughly 23 Composer packages, which is an acceptable price for a well-documented,
maintained tool that both contributors can rely on without project-specific
onboarding.

## Decision

Use Phinx (`robmorgan/phinx`) as the migration runner.

### Migration Files: `/database/migrations/`

Migration files live in `/database/migrations/` and are generated by Phinx with
timestamp-based names (`YYYYMMDDHHMMSS_ClassName.php`). The timestamp prefix
establishes execution order and prevents conflicts between contributors working
in parallel. Generated files are never renamed, because the filename is part of
how Phinx identifies the migration.

Migrations are written using Phinx's `change()` method and native table-builder
API (`createTable()`, `addColumn()`, `addForeignKey()`, `addIndex()`,
`renameColumn()`, etc.), which Phinx auto-reverses on rollback. Raw SQL via
`$this->execute()` is avoided because it cannot be auto-reversed.

Existing FIX scripts that are purely schema migrations (not admin utilities) are
candidates for conversion to this format as part of the issue #595 restructuring.
Contributor-facing guidance lives in
[`database/migrations/README.md`](../../../database/migrations/README.md).

### Tracking Table: `phinxlog`

Phinx creates and maintains its own `phinxlog` table, storing one row per applied
migration (version, name, start/end time). A migration recorded in `phinxlog` is
never applied again. A migration that fails is not recorded, so re-running after
a fix safely retries it. The table is managed by Phinx and is not modified
manually.

### Commands

The workflow is exposed through Composer wrappers:

```bash
composer migrate            # Apply all pending migrations
composer migrate:status     # Show pending and applied migrations
composer migrate:dry-run    # Preview pending migrations without applying
composer migrate:rollback   # Roll back the most recent migration
```

New migrations are scaffolded directly with `vendor/bin/phinx create MigrationName`,
which generates the timestamped file.

### Deployment Integration

Migrations run as an explicit CLI step during deployment (`composer migrate`),
documented in `docs/development/DEPLOYMENT.md`. Phinx exits non-zero on failure,
making it suitable for a deployment script that checks exit codes. Migrations are
never triggered by a web request, admin panel action, or application
initialization sequence.

### What This Replaces

This mechanism replaces the one-time schema-migration responsibility of the FIX
script system. Admin utilities and ad-hoc diagnostic scripts are not affected by
this ADR; their restructuring is tracked separately in issue #595.

## Consequences

### Positive

- **Rollback support.** Migrations written with `change()` and Phinx's native

  table-builder API are auto-reversed by `composer migrate:rollback`. No hand-
  written reverse migration is required for the common case.

- **Community documentation.** Both contributors can learn the tool from Phinx's

  own documentation rather than from a project-specific runner and README. This
  is the decisive benefit given a second, occasional contributor.

- **`--dry-run` preview.** `composer migrate:dry-run` prints the pending

  migrations without applying them, allowing a change to be reviewed before it
  touches the schema.

- **`status` command.** `composer migrate:status` shows which migrations are

  applied and which are pending in the current environment at a glance.

- **Deterministic ordering and conflict prevention.** Timestamp-based filenames

  give a total ordering and reduce cross-contributor conflicts to near zero.

- **Deployment safety.** Migrations run via CLI at deploy time, not via browser.

  There is no web-facing entry point and therefore no associated attack surface.
  Phinx exits non-zero on failure, and a failed migration is not recorded, so
  re-running after a fix safely retries it.

### Negative

- **Composer dependency.** Phinx adds roughly 23 Composer packages, each with its

  own version constraints and upgrade path. This is the cost accepted in exchange
  for rollback support, documentation, and conflict-safe ordering. It is a
  larger dependency footprint than a bespoke runner would carry, but far smaller
  than a full application framework.

- **Developers must remember the deployment step.** There is no automatic

  migration on application startup. A deployment that skips `composer migrate`
  leaves the schema out of sync with the code. This is mitigated by the
  deployment checklist.

### Risks

**Developer forgets to run migrations during deployment** (Medium likelihood,
Medium impact). Add the `composer migrate` step to the deployment checklist;
`composer migrate:status` makes drift easy to detect.

**DDL migration cannot be wrapped in a transaction** (Low likelihood, Medium
impact). MySQL issues an implicit commit on DDL, so a partial DDL change is not
atomic. Document non-transactional operations in the migration file header;
a failed migration is not recorded in `phinxlog`, so the fixed migration can be
re-run.

## Alternatives Considered

### Custom Lightweight Runner

Build a ~100-line PHP runner over `mysqli` with a `schema_migrations` tracking
table and a `--dry-run` flag, avoiding any Composer dependency. This was the
approach proposed in the earlier draft of this ADR.

**Rejected because:**

- The arrival of a second, occasional contributor changed the trade-off. A

  bespoke runner requires project-specific onboarding, whereas Phinx is
  self-documenting for anyone who reads its docs.

- Rollback support is genuinely useful and the custom runner offered none;

  adding a `down()` mechanism to it would have reinvented a slice of Phinx.

- Timestamp-based conflict prevention and idempotent tracking come for free with

  Phinx. Building and maintaining the equivalent is application code the project
  would own indefinitely.

Taken together, the second contributor plus the rollback need tipped the balance
to Phinx despite the dependency cost.

### Heavyweight Migration Frameworks (Doctrine Migrations, Laravel Migrations)

Adopt a larger, more feature-rich migration framework via Composer.

**Rejected because:**

- Doctrine Migrations assumes the Doctrine ORM/DBAL stack; Laravel Migrations

  assumes the Laravel/Illuminate ecosystem. Neither is present in this UserSpice
  application, so both pull in far more than migration support.

- Phinx provides the required features (rollback, `status`, `--dry-run`,

  timestamp ordering) as a standalone tool without imposing an ORM or framework.

### Web-Accessible FIX Scripts (Status Quo)

Continue applying schema changes via web-accessible PHP scripts in `/FIX/`.

**Rejected because:**

- Schema-changing scripts should not be accessible via browser. A logged-in

  admin session check is a weaker boundary than no web-facing entry point at all.

- The FIX system provides no atomic tracking that a given schema change has run

  exactly once. `fix_script_runs` records a run attempt but does not prevent
  re-execution.

- The 243-line FIX script template is over-engineered for a file whose sole

  purpose is "execute this DDL once."

### Raw Manual SQL via phpMyAdmin or CLI

Execute schema changes manually without any runner or tracking.

**Rejected because:**

- No tracking: there is no record of which changes have been applied to which

  environment. Reproducing the schema on a fresh installation requires reading
  through deployment notes.

- No idempotency: running a `CREATE TABLE` statement twice raises an error;

  running an `ALTER TABLE` that has already been applied may silently corrupt
  data or fail.

- Error-prone: manual execution is the most common source of "works on staging,

  broken in production" incidents.

### Keep FIX Scripts but Restrict to CLI

Adapt the existing FIX script mechanism to run via CLI instead of web, without
introducing a separate migration runner.

**Rejected because:**

- The FIX template provides a two-phase web UI with progress logging; stripping

  that down to CLI output requires rewriting most of the template, at which point
  a new, purpose-built runner is simpler.

- The FIX system does not provide timestamp-ordered execution of multiple

  migrations in a single command. Each FIX script is a standalone invocation;
  a deployment would require running them individually in the correct order.

- Keeps the FIX template as the migration interface, which conflates admin

  utilities with schema migrations and makes it harder to distinguish "must run
  at deployment" from "run when an admin requests it."
