# Elan Registry v2.27.0 Release Notes

**Release Date:** 2026-07-19
**Type:** Minor Release - Security Hardening

## Required Actions After Deployment

- **Run `01-Rename-Legacy-Image-Files.php`** (Admin → Maintenance): Renames 9 image files whose filenames contain characters now blocked by the #1307 allowlist (parentheses, `+`, brackets, Unicode), and removes 2 stale DB entries that have no file on disk. Safe to re-run — already-renamed files are detected and skipped. Run on `test.elanregistry.org` first, verify car pages load, then run on `elanregistry.org`. After production run, purge the Cloudflare cache for affected image URLs.

## User-Facing Changes

### Improvements

- **Contact Owner Privacy** ([#1322](https://github.com/elan-registry/registry/issues/1322)): Contact-owner page now shows only the owner's first name, consistent with the published privacy policy.

## Admin-Facing Changes

### Improvements

- **Owner Privilege Escalation Guard** ([#1232](https://github.com/elan-registry/registry/issues/1232)): Removed `active` and `permissions` from `Owner::extractUserFields()` allowlist; changed the `validateAndSanitizeFields()` default case to drop unknown fields instead of passing them through. Prevents any future endpoint from accidentally writing these privilege-controlling columns via the general profile-update path.
- **Server Hardening** ([#1242](https://github.com/elan-registry/registry/issues/1242)): Blocked web access to PHPUnit config, PHPStan config, and npm manifests; disabled HTTP TRACE; added Permissions-Policy header opting out of geolocation, camera, microphone, and payment APIs.
- **CSP Tightening** ([#1326](https://github.com/elan-registry/registry/issues/1326)): Added `form-action 'self'` to the Content Security Policy (closing a form-hijacking gap that `default-src` doesn't cover); removed `unsafe-eval` from `script-src` after verifying no custom JS uses `eval()` or `new Function()`.
- **Backup Authorization** ([#1308](https://github.com/elan-registry/registry/issues/1308)): Backup and restore operations now require Administrator role; editor accounts are explicitly rejected.
- **Admin Script CSRF Hardening** ([#1308](https://github.com/elan-registry/registry/issues/1308)): Destructive admin maintenance scripts now require POST + CSRF token. A new shared `fix-script-core.php` centralises the auth gate, start-form, and "Close Window" button so all current and future scripts inherit the pattern automatically. "Return to Admin Console" buttons replaced with "Close Window" (closes the window and reloads the maintenance tab).
- **Admin Script Role Guard** ([#1338](https://github.com/elan-registry/registry/issues/1338)): `admin_script_exec_requested()` in `fix-script-core.php` now enforces Administrator role in addition to POST + CSRF, so editor accounts cannot trigger destructive execute paths even if page-level permissions allow them to reach the script. Denied attempts are logged to the security audit log. The thumbnail maintenance script's batch continuation path (which bypasses the shared gate) receives its own `isAdmin()` guard. Also removes pre-auth `display_errors` from the thumbnail script and hardens its error reporting.
- **Login Audit Logging** ([#1243](https://github.com/elan-registry/registry/issues/1243)): Confirmed that UserSpice rate-limiting is active and enforcing lockout (5 failures per account / 5 min; 20 per IP / 15 min). Added `loginFail` and `loginSuccess` security-category log entries via hooks, mirroring the logging the upstream framework ships with commented out.
- **Admin User Detail XSS Fix** ([#1306](https://github.com/elan-registry/registry/issues/1306)): HTML-escaped every profile and car field rendered in the admin "manage user" hook. Closes a stored-XSS path where an owner's own `city`/`state`/`country` (or car chassis metadata) could execute in an admin session. Also guards the profile block against missing rows with a "No profile data" placeholder instead of PHP warnings.
- **Public API PII Stripping** ([#1305](https://github.com/elan-registry/registry/issues/1305)): Removed `email`, `lname`, `vericode`, and `last_verified` from the `ALLOWED_COLUMNS` allowlist in `CarDataTablesService`. Replaced `SELECT *` with explicit column lists in both `getDataTablesData()` (car list and factory endpoints) and `CarRepository::getHistory()` (history endpoint), so future schema additions cannot silently expose owner PII or internal fields to unauthenticated callers. CSRF-only authentication on these endpoints remains unchanged; the fix is data minimization at the query layer.
- **DataTables Stored XSS Fix** ([#1304](https://github.com/elan-registry/registry/issues/1304)): Applied `DataTables.render.text()` to every text column in the car list, factory data, and car history tables. Previously, raw database values were inserted via `innerHTML`, allowing a malicious `color` or `chassis` value to execute as XSS in any logged-in viewer's session. Also added a character-set allowlist to `ChassisValidator` (digits, letters, `/`, `-` only) that runs before the admin override branch, so override cannot be used to store HTML-injection payloads.
- **Image Filename Allowlist** ([#1307](https://github.com/elan-registry/registry/issues/1307)): Validated uploaded image filenames against the `img_[32hex].(jpg|png|gif|webp)` format produced by `generateSecureFilename()` before any `glob()` or filesystem operation. Closes two paths: (1) glob hijack — a client-supplied `filenames=*` value that matched every in-flight upload in the shared temp directory; (2) traversal metadata oracle — filenames like `../../../etc/passwd` that triggered `is_file()`/`filesize()`/`exif_imagetype()` on arbitrary paths. `generateSecureFilename()` is now a method on `CarImageProcessor` and shares `ALLOWED_EXTENSIONS` with the validation logic so the two cannot drift apart. The fix script `01-Rename-Legacy-Image-Files.php` migrates the 11 pre-existing DB rows that contain legacy filenames now rejected by the allowlist (see Required Actions).
- **Concurrency Guard — Image Removal and Car Merge/Delete** ([#1311](https://github.com/elan-registry/registry/issues/1311)): Closed three race-condition bugs in car data mutations. Image removal now uses compare-and-swap (`WHERE id = ? AND image = ?`) so two concurrent removals cannot silently overwrite each other. The admin merge transaction acquires a `SELECT … FOR UPDATE` row lock on the source car before any structural operations, preventing a concurrent delete from producing a corrupted audit trail. Car delete now checks rows-affected and throws `CarNotFoundException` on a zero-row result rather than logging a false success for an already-deleted car.
- **Admin Merge Migrated to Service Layer** ([#1349](https://github.com/elan-registry/registry/issues/1349)): The bespoke merge transaction in `admin/index.php` has been replaced with a `Car::merge()` delegation. The admin page now uses the same `CarAdministrationService::merge()` path as all other callers, eliminating duplicated transaction management code and ensuring the `SELECT … FOR UPDATE` concurrency guard (from #1311) is active on the admin merge path automatically. The audit trail operation code changes from `DUPLICATE`/`NEWOWNER` to `MERGE`; the admin flash message remains unchanged.
- **Fix-Page-Permissions Security and Error Hardening** ([#1354](https://github.com/elan-registry/registry/issues/1354)): Added `isAdmin()` guard to the execute-form path in `21-Fix-Page-Permissions.php` (the AJAX paths already had the guard; the form submit path did not). Moved `analyzePermissions()` inside the outer `try` block so a DB failure is caught and logged rather than producing an opaque fatal error. Added `$scriptFailed` flag so the "Script completed" success message is suppressed when processing fails.
- **Fix Script Error Handling** ([#1353](https://github.com/elan-registry/registry/issues/1353)): `buildAffectedList()` in `01-Rename-Legacy-Image-Files.php` now runs inside a try/catch. A DB failure previously produced an opaque PHP fatal error with no log entry; it now logs via `logger()` and shows a readable alert to the authenticated admin.
- **Fail-Open Guard Closure** ([#1312](https://github.com/elan-registry/registry/issues/1312)): Closed two fail-open paths that silently returned permissive answers on errors. The chassis-availability check now returns a server error (never "available") when the DB query fails. The transfer-request endpoint throws a user-visible error (never "No car found") on DB failure. The image-upload path guard now rejects explicitly when `realpath()` returns `false` for a missing or misconfigured image directory, closing a PHP type-coercion bug where `strpos($path, false)` returned `0` and silently skipped the traversal guard entirely.
- **Admin JS Extraction** ([#1327](https://github.com/elan-registry/registry/issues/1327)): Moved all inline `<script>` blocks from admin templates to external cached JS files, eliminating `unsafe-inline` from the admin panel's effective script execution surface. The main admin JS file is renamed `admin-core.js` and gains reusable delegated event handlers (`[data-dismiss-target]`, `[data-switch-tab]`, `[data-confirm-form]`, `[data-hover-row]`, `[data-feedback-link]`) that replace a dozen single-purpose named functions. Per-owner data needed by the owner-profile form is now emitted via a structured `window.elanOwnerProfileData` island; all other tab scripts derive endpoint URLs from the existing `window.elanUrlRoot` global, eliminating PHP-injected URL strings. Six new tab-specific JS files (`design-system`, `tab-manage-cars`, `tab-account-cleanup`, `tab-owner-mgmt`, `tab-settings`, `load-owner-profile`) are minified through the esbuild pipeline and covered by ESLint. Prerequisite for removing `unsafe-inline` from `script-src` (issue #1328).
- **Car Edit DOM XSS Fix** (CodeQL): `car-edit.js` used `$('#car_id').html(value)` to display a car ID, injecting the field value as HTML rather than text. Changed to `.text()`. No exploitable path existed (car IDs are integers), but the pattern was a genuine DOM XSS CWE-79 finding that CodeQL correctly flagged.
- **User-Facing JS Extraction + CSP `unsafe-inline` Removal** ([#1328](https://github.com/elan-registry/registry/issues/1328)): Extracted all inline `<script>` blocks from seven user-facing templates into five new external cached JS files (`car-list.js`, `factory-list.js`, `car-details-map.js`, `car-edit.js`, `contact-form.js`). Two contact-form pages share a single JS file via a `data-api-url` attribute on the form element. Duplicate PHP config logic (thumbnail sizes) was extracted to a shared partial (`includes/elan-config-island.php`). All raw PHP echoes into JS strings were replaced with `json_encode()` calls during migration. `'unsafe-inline'` is removed from `script-src`; a per-request nonce (`$userspice_nonce = base64_encode(random_bytes(16))`) is generated in `security_headers.php` and wired into both the CSP header and every data-island `<script nonce="...">` block. Four SHA-256 hashes are kept as belt-and-suspenders for static upstream scripts. Three bugs surfaced and fixed during nonce wiring: (1) a `?>` closing tag inside a `//` comment in `security_headers.php` terminated PHP mode and caused the entire remainder of the file to render as literal text on every page; (2) a `$usespice_nonce` typo (missing 'r') in `footer.php` (3 occurrences) and `account.php` (2 occurrences) caused all nonces generated by those pages to evaluate to empty string, blocking inline scripts site-wide; (3) two `<script>` blocks on the home page (`index.php`) lacked nonce attributes entirely and were blocked by the new CSP. Playwright CSP tests pass. ADR-007 updated to reflect the nonce strategy adoption.
